# Deferred Gates, Risks, and Stop Triggers

## Separately gated

- Perfect Essence removal/capacity policy and six prepared Perfect rows.
- Crystallisation Omens and every other Omen/modifier layer.
- Omen catalogue reconciliation against current external sources.
- Alchemy four-add behavior.
- Fracturing, including the disputed revealed-Desecrated rule.
- Jawbone, Desecrated placeholders, Reveal, Lich constraints, and PD-013.
- Longer chains, route planning, optimizer/economics/advice, and public numeric output.
- SOURCE/PROVENANCE, broader MML, or PD-013 closure.

## Key risks

| Risk | Control |
|---|---|
| Executor silently performs an ordinary weighted add | Exact terminal must contain the declared guaranteed `mod_id`; no candidate draw is allowed |
| Rarity mutates before final legality is known | Isolated working copy and atomic commit |
| Data files disagree | Resolver/compiler cross-file assertions, fail closed |
| Crafted capacity bypassed | Shared state validation is load-bearing |
| Eight copied handlers drift | One executor; row metadata selects output |
| Perfect behavior leaks into Greater core | Explicit operation-id allowlist and negative tests |
| External catalogue drift becomes accepted truth | Package records evidence only; data changes need a separate gate |

## Stop triggers for later implementation

- any selected row no longer matches trusted source/item applicability;
- operation/output registries disagree;
- implementation requires a new probability assumption;
- implementation requires changing accepted ordinary-add, M40-A, capacity, or fractured semantics;
- Perfect/Omen/Alchemy behavior becomes necessary;
- result cannot remain one shared executor;
- runtime admission would exceed the eight authorized rows;
- any public numeric, optimizer/economics/advice, automation, or boundary closure enters scope.
