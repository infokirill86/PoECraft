# Recommended Broad M45-A Implementation Batch

M45-A should be one audited implementation batch for the ten clean-core Omens listed in `05_SELECTED_M45_BOUNDARY_AND_ROUTE_CONNECTION.md`.

## Why batching is safe

- All effects modify already accepted operations; no new base operation is introduced.
- Four explicit effect dimensions can be independently reconstructed and tested.
- The accepted resolver is the only compilation seam.
- The accepted exact and seeded-MC executors can test the same resolved plan.
- Side filters and Whittling operate on existing canonical pools rather than new data sources.
- Greater Exaltation can reuse the accepted atomic sequential-add workspace and pool-rebuild pattern.

## Required M45-A evidence

1. Omen inventory/status reconciliation and a validator separating catalogue, availability, and runtime admission.
2. Resolver fail-closed behavior for every Omen outside the ten-row allowlist.
3. Direct-versus-resolver and one-step M43-A parity for every admitted combination.
4. Prefix/suffix pool proofs for Exalted, Annulment, Chaos removal, and Perfect Essence feasible removal.
5. Whittling proof: eligibility/fractured exclusion first, minimum modifier level second, uniform project-policy tie handling last.
6. Greater Exaltation proof: two sequential accepted adds, branch-state rebuild, atomic rollback, no partial result.
7. Exact mass conservation, canonical terminal aggregation, seeded replay, diagnostics, and negative controls.
8. Full regression and semantic-fingerprint delta restricted to the pinned Omen admission/data changes.

## Build-gate pins still required

The later ChatGPT/User implementation gate must explicitly ratify Greater Exaltation’s sequential-draw/atomic project model and the exact ten-row allowlist. If that ratification is withheld, split Greater Exaltation out and implement the nine filter/selector Omens first.

This package does not authorize either implementation path.
