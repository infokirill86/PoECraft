# Separate Gates, Risks, and Stop Conditions

## Separately gated

- `light`: depends on Desecrated modifier semantics.
- Jawbone/Reveal Omens: depend on unaccepted Jawbone/Reveal runtime and PD-013-sensitive staged state.
- Alchemy and Coronation side Omens: historical/drop-disabled status and exact generation contract require a version-policy gate.
- Greater Annulment: two-removal selection, aggregation, atomicity, and insufficient-target behavior require a mechanics gate.
- Homogenising and any catalogue entries absent from canonical repo data: inventory/source reconciliation first.
- Fracture, Essence repeat/replacement/capacity, Desecrate/Jawbone/Reveal, longer/unbounded route generation, planner, optimizer, economics, advice, and public numeric release.

## Main risks

| Risk | Control |
|---|---|
| Treating `data/omens.yaml` presence as admission | Separate validated runtime-admission status |
| Version drift or legacy items silently entering current scope | Explicit availability/version status and User gate |
| Applying filters at the wrong stage | Per-family load-bearing pool proof |
| Parallel legality/removal implementations | Require accepted canonical pool builders |
| Combining incompatible Omens | Dimension-based compatibility matrix, fail closed |
| Greater Exaltation exposing a one-add partial state | Private working state and atomic commit |
| Modifier compiler turning into a route planner | One currency invocation only; no conditions, retries, ranking, or selection |

## Stop conditions for later implementation

Stop if source reconciliation changes the intended ten-row set, if a clean-core effect requires a new base operation or new canonical pool, if exact behavior cannot be specified before tests, if runtime permission would be inferred from catalogue/project-scope activity, or if any boundary closure/public output/planner behavior enters scope.
