# Selected M45 Boundary and Route Connection

## Selected design boundary

M45 designs an independent, single-operation Omen modifier layer:

`accepted currency request + admitted Omen set + current item state -> resolved accepted operation plan`

The resolver remains a compiler for one operation invocation. M43-A may later execute the resulting operation inside a caller-supplied fixed sequence, but M45 does not choose, compare, or recommend sequences.

## Proposed clean-core allowlist for a later M45-A gate

| Omen | Target accepted operation family | Effect dimension |
|---|---|---|
| `greater_exaltation` | Exalted-like add | atomic add-count override to two |
| `sinistral_exaltation` | Exalted-like add | prefix-only add pool |
| `dextral_exaltation` | Exalted-like add | suffix-only add pool |
| `sinistral_annulment` | Annulment | prefix-only removable pool |
| `dextral_annulment` | Annulment | suffix-only removable pool |
| `sinistral_erasure` | Chaos-like | prefix-only removal pool; add stage unchanged |
| `dextral_erasure` | Chaos-like | suffix-only removal pool; add stage unchanged |
| `whittling` | Chaos-like | minimum modifier-level eligible removal; tied minima use the existing project-model uniform policy |
| `sinistral_crystallisation` | accepted Perfect Essence rows | prefix-only feasible removal pool |
| `dextral_crystallisation` | accepted Perfect Essence rows | suffix-only feasible removal pool |

No row above is admitted by this design package.

## Direct route connection

The physical-quarterstaff route depends on controlling whether valuable prefix/suffix space is added or removed. Side-directed Exalted, Annulment, Chaos removal, and Perfect Essence removal therefore affect the actual route more directly than modelling creation of the already-purchased fractured base. Whittling adds targeted removal behavior without introducing a new currency primitive.

Greater Exaltation reuses the accepted sequential-add and atomic-workspace pattern. It is useful but its internal two-add sampling/atomicity remains a project-model contract to be explicitly ratified in the later implementation gate, not a server-truth claim.
