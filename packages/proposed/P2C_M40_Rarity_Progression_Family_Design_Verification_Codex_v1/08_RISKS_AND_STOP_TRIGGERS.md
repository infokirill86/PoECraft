# Risks and stop triggers

| Risk | Consequence | Control |
|---|---|---|
| Pool built at input rarity | Transmutation may see no capacity; Regal may use magic rather than rare legality | Target-rarity working-state proof and negative control. |
| Rarity committed before add succeeds | Partial upgraded item after failure | Atomic isolated state and rollback/no-consumption tests. |
| One function copied per currency | Divergent weights, MML, and failure behavior | One data-driven one-add plan/executor. |
| Augmentation side is chosen separately | Distorts combined legal-pool weights | Let ordinary capacity filtering remove the full side. |
| `active_in_current_simulation` treated as permission | Unaccepted rows execute | Require accepted `runtime_admission_status` plus resolver registration. |
| MML threshold treated as full mechanics closure | Overclaims evidence | Preserve project-model/source-open label and separate broader MML gate. |
| Source registry points to the wrong contextual patch | Weak provenance chain | Add official 0.3.0 reference during a later authorized evidence update. |
| Batch expands into Alchemy/Essence/Omen behavior | Mixes unrelated mechanics | Stop and return to ChatGPT/User. |

## Stop triggers for any later implementation

- trusted-source conflict or changed thresholds;
- ambiguity about target-rarity pool legality that cannot be resolved by the accepted project-model inference;
- any requirement to change accepted ordinary-add, MML, exact, or MC kernels;
- unsupported operation/modifier combination does not fail closed;
- new operation admission beyond the ten named rows;
- public numeric output, optimizer/economics/advice, automation, or boundary closure enters scope.
